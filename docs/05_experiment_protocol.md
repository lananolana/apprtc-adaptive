# Протокол эксперимента baseline vs adaptive

Это **операционный чек-лист** для проведения экспериментов под Главу 3.
Не для ВКР — для тебя. Цель: получить CSV-файлы, по которым я наполню
§3.3 фактическими цифрами и графиками.

Эксперимент состоит из трёх частей:

| Часть | Что | Сколько прогонов | Где | Сколько времени |
|---|---|:---:|---|:---:|
| A | Контролируемые NLC-прогоны | 10 | Одна машина, две вкладки Chrome | ~45 мин |
| B | Outage на двух машинах | 2 | Две машины, одна сеть Wi-Fi | ~15 мин |
| C | Полевой кейс PT↔MSK | 1–2 | Две машины в разных странах | ~25 мин + договориться с другом |

**Итог:** ~1,5 часа активного времени, 13–14 CSV-файлов.

# Часть A. Контролируемые прогоны на одной машине через NLC

Используем Apple Network Link Conditioner для воспроизводимых сетевых
условий. Эти 10 прогонов — научный костяк §3.3 (§3.3.1–§3.3.4).

## Что понадобится

1. Mac с актуальным Chrome.
2. **Network Link Conditioner** (NLC). Если ещё не установлен:
   - Открой [Apple Developer Downloads](https://developer.apple.com/download/all/) (нужен Apple ID).
   - Скачай **«Additional Tools for Xcode»** (свежий, под твою macOS).
   - В скачанном `.dmg` открой `Hardware/Network Link Conditioner.prefPane` — двойной клик ставит его в System Settings.
   - В **System Settings → Network Link Conditioner** появится новая панель.
3. Прототип на проде: <https://webrtcvkr.nomorepartiessite.ru> — должен открываться, healthz возвращать «ok».

## Однократная настройка NLC: создание профилей

В System Settings → Network Link Conditioner → **Manage Profiles → +** создай 4 кастомных профиля. Имена полей в таблице ниже **в точности** совпадают с интерфейсом NLC, порядок столбцов тот же, в каком NLC их рисует в окне настройки.

| Имя профиля | Downlink Bandwidth | Downlink Packets Dropped | Downlink Delay | Uplink Bandwidth | Uplink Packets Dropped | Uplink Delay | DNS Delay |
|---|:---:|:---:|:---:|:---:|:---:|:---:|:---:|
| `3g` | 750 Kbps | 0,5 % | 100 ms | 750 Kbps | 0,5 % | 100 ms | 0 ms |
| `edge` | 250 Kbps | 1 % | 200 ms | 250 Kbps | 1 % | 200 ms | 0 ms |
| `lossy` | (пусто, без огран.) | 5 % | 50 ms | (пусто) | 5 % | 50 ms | 0 ms |
| `severe` | 100 Kbps | 15 % | 400 ms | 100 Kbps | 15 % | 400 ms | 0 ms |

Несколько важных замечаний:

- **«Downlink Packets Dropped»** в UI NLC — это то же самое, что
  packet loss в нашей таблице 3.2 в ВКР. Я переименовала колонки под
  фактический интерфейс, чтобы не путать.
- **Задержка указывается в одну сторону.** Полный RTT, который потом
  увидит WebRTC через `getStats`, ≈ uplink + downlink delay. Значения
  поделены пополам, чтобы суммарный RTT совпал с заявленным в ВКР
  (для `3g` → 200 мс, для `edge` → 400 мс, для `lossy` → 100 мс, для
  `severe` → 800 мс).
- **DNS Delay = 0 ms.** Этот параметр задерживает разрешение
  DNS-имён, но не влияет на уже установленную WebRTC-сессию (медиа
  идёт через UDP/SRTP по согласованным IP, без обращения к DNS). Мы
  ставим 0, чтобы ничего не накладывалось на установление сессии и
  на здоровье HTTPS-соединения с сигналинг-сервером.
- **Bandwidth — выпадающий список с единицами.** Убедись, что в нём
  выбран **Kbps** (не Mbps и не Bps); по умолчанию там Mbps, легко
  пропустить.

Профиль `good` — это просто состояние OFF (NLC выключен). Отдельно
создавать его не нужно.

## Подготовка

1. Закрой Slack, Telegram-десктоп, iCloud sync, торренты — всё, что грузит сеть.
2. Запусти Chrome (отдельный профиль / окно incognito — для чистоты).
3. Открой DevTools (F12) на случай отладки.

## Один прогон (повторяется 10 раз)

Шаги ниже — для одной комбинации `<профиль> × <режим>`. Всего комбинаций 10: 5 профилей × 2 режима.

### Старт сессии

1. Открой <https://webrtcvkr.nomorepartiessite.ru> в **первой** вкладке.
2. Нажми «Войти / создать», разреши камеру и микрофон.
3. Нажми «Копировать ссылку».
4. Открой **новую вкладку** в этом же окне, вставь ссылку, нажми Enter.
5. Во второй вкладке нажми «Войти / создать». Появится локальное видео и через 1–3 секунды — удалённое.
6. В обеих вкладках состояние ICE = `connected`, метрики бегут.

### Выбор режима

В **первой** вкладке выбери `baseline` или `adaptive`.

### Запись

1. В поле profile впиши `good` / `3g` / `edge` / `lossy` / `severe`.
2. Нажми **● Старт записи** → красная кнопка «⏺ Стоп и скачать».
3. **30 секунд (pre)** — NLC выключен, нормальные условия.
4. **На 30-й секунде:** включи NLC с нужным профилем, нажми **Mark** в прототипе.
5. **60 секунд (load)** — наблюдай. В `adaptive` будут видны переключения политики.
6. **На 90-й секунде:** выключи NLC, обнови profile-поле на `back-to-good`, нажми **Mark**.
7. **30 секунд (post)** — наблюдай восстановление.
8. **Через 30 секунд:** ⏺ Стоп и скачать. CSV скачается с автоименем.

### Завершение

Нажми «Завершить», закрой обе вкладки.

## Программа Части A — 10 прогонов

```
1.  baseline + good     →  run-good-baseline-….csv
2.  baseline + 3g       →  run-3g-baseline-….csv
3.  baseline + edge     →  run-edge-baseline-….csv
4.  baseline + lossy    →  run-lossy-baseline-….csv
5.  baseline + severe   →  run-severe-baseline-….csv
6.  adaptive + good     →  run-good-adaptive-….csv
7.  adaptive + 3g       →  run-3g-adaptive-….csv
8.  adaptive + edge     →  run-edge-adaptive-….csv
9.  adaptive + lossy    →  run-lossy-adaptive-….csv
10. adaptive + severe   →  run-severe-adaptive-….csv
```

# Часть B. Outage на двух машинах (RecoveryManager)

Здесь Wi-Fi реально выключается на одной из машин — это самый честный
тест для RecoveryManager. Без эмулятора, без NLC.

## Что понадобится

Два устройства на твоей стороне с современным Chrome:
- **A:** Mac (основной, на нём ведём запись).
- **B:** второй девайс — лучше всего ноутбук, но Chrome на Android / iPad тоже работает.

Оба в одной сети Wi-Fi (домашняя).

## Подготовка

1. **На устройстве A** открой <https://webrtcvkr.nomorepartiessite.ru>, нажми «Войти / создать», скопируй ссылку.
2. **На устройстве B** открой ту же ссылку, нажми «Войти / создать».
3. Дождись `connected` на обоих устройствах.

## Прогон B1 — outage в baseline

Делается **с устройства A** (там запись).

1. Убедись, что в первой вкладке выбран режим **baseline**.
2. Впиши profile = `outage`.
3. **● Старт записи** на A.
4. **30 сек** — ничего не трогать.
5. **На 30-й секунде:** на устройстве **B** выключи Wi-Fi (System Settings → Wi-Fi → OFF, или Airplane Mode).
6. **5 сек** держи Wi-Fi выключенным. На устройстве A видео должно «зависнуть», в правом верхнем углу через пару секунд появится оверлей «Собеседник вышел из звонка» (потому что сервер заметит отвал WebSocket быстрее, чем WebRTC) или фриз/recovery.
7. **На 35-й секунде:** на устройстве **B** включи Wi-Fi обратно.
8. **Наблюдай ~60 секунд.** В **baseline** связь автоматически не вернётся — ICE-state на A будет `failed`, видео не возобновится без ручного действия.
9. **⏺ Стоп и скачать** → `run-outage-baseline-….csv`.
10. На обоих устройствах нажми «Завершить».

## Прогон B2 — outage в adaptive

Те же шаги, только в первой вкладке устройства A выбран **adaptive**.

Ожидаемое поведение:
- На 30-й секунде Wi-Fi выключается на B.
- На устройстве A через 3 секунды (`disconnectTimeoutMs`) RecoveryManager триггерится: появится тост «Соединение восстанавливается. Попытка 1 из 3», красный оверлей.
- На 35-й секунде Wi-Fi включается на B → ICE restart должен установить связь заново.
- Через 5–15 секунд должно появиться «Связь восстановлена», видео возвращается.

CSV → `run-outage-adaptive-….csv`.

## Опционально: запись с обеих сторон

Если есть возможность, на устройстве B параллельно тоже нажми «● Старт записи» — получишь два CSV для одного события («как это выглядело глазами каждого пира»). Это сильно усилит §3.3.5. Имя файла на B: добавь суффикс `-side-b` руками после скачивания.

# Часть C. Полевой кейс PT↔MSK

Реальная межконтинентальная WebRTC-сессия — один из самых сильных
аргументов в защите. Здесь нет лабораторного контроля, есть living experiment.

## Что понадобится

- Договорённость с другом в Москве (на твой выбор).
- Согласованное время (15–20 минут на разговор).
- У обоих современный Chrome.
- Канал координации — Telegram/WhatsApp/etc, чтобы говорить «начали запись», «попроси я подвигаюсь активнее», «всё, конец».

## Подготовка

1. **Ты (Португалия):** заходишь на <https://webrtcvkr.nomorepartiessite.ru>, создаёшь комнату, копируешь ссылку, кидаешь другу.
2. **Друг (Москва):** открывает ссылку, нажимает «Войти / создать», принимает камеру/микрофон.
3. У обоих появляется удалённое видео.

## Прогон C1 — adaptive

1. У тебя в режиме `adaptive`. (У друга режим не критичен — это его клиент, его контроллер; для эксперимента важно, что **на твоей стороне** ведётся запись и работает adaptive.)
2. profile = `field-pt-msk-adaptive`.
3. **● Старт записи**.
4. **15 минут разговора** — поговорите о чём угодно. Желательно с активной мимикой, иногда подвигайтесь — это нагружает кодек и провоцирует deltas. Если в какой-то момент сеть пойдёт волной (часто бывает на интернациональных линках) — будут видны срабатывания контроллера.
5. **⏺ Стоп и скачать** → `run-field-pt-msk-adaptive-….csv`.

## Прогон C2 — baseline (опционально, если хватит времени и терпения друга)

То же самое, но в режиме `baseline`. Сравнение можно будет провести по статистике RTT/loss/bitrate за равные интервалы.

profile = `field-pt-msk-baseline`.

## Опционально: запись со стороны друга

Если друг готов нажать пару кнопок:
- На его стороне в DevTools открой Console и попроси скинуть тебе вывод `pc.getStats()` пару раз, или
- Попроси его тоже нажать «● Старт записи» и прислать тебе CSV (имя `run-field-pt-msk-{mode}-side-b-….csv`).

Это даёт «вид с обеих сторон» одного интернационального звонка — отличный материал для §3.4.

# После всех прогонов

1. Собери все CSV в одну папку (например, `~/Desktop/webrtc-runs/`):
   ```
   run-good-baseline-….csv
   run-3g-baseline-….csv
   ...
   run-outage-baseline-….csv
   run-outage-adaptive-….csv
   run-field-pt-msk-adaptive-….csv
   (опционально) run-field-pt-msk-baseline-….csv
   ```
2. Zip-ни папку или перетащи мне файлы в чат.
3. Я прогоню через анализ-скрипт, получу сводные таблицы и графики, вставлю в §3.3 и §3.4.

## Если что-то не работает

- **Звонок не устанавливается** — DevTools Console на одной из сторон, скинь логи `[app]`.
- **Дашборд показывает нули** — `getStats()` пуст. Дождись `connected`, переподключись.
- **NLC «не действует»** — выключи и снова включи переключатель ON в System Settings.
- **Запись пустая** — нужно сначала установить соединение, потом нажать «Старт записи».
- **На двух устройствах: трафик не идёт** — проверь, что оба в одной сети, или что прямое соединение возможно. STUN-сервер `stun.l.google.com:19302` обычно справляется.
- **Кнопки в интерфейсе нет** — Cmd+Shift+R, CSS/JS могли залипнуть в кэше.
