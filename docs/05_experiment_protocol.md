# Протокол эксперимента baseline vs adaptive

Это **операционный чек-лист** для проведения экспериментов под Главу 3.
Не для ВКР — для тебя. Цель: получить 10 CSV-файлов, по которым я
наполню §3.3 фактическими цифрами.

Активное время: 30–40 минут.

---

## Что понадобится

1. **Mac** с актуальным Chrome.
2. **Network Link Conditioner** (NLC). Если ещё не установлен:
   - Открой [Apple Developer Downloads](https://developer.apple.com/download/all/) (нужен Apple ID).
   - Скачай **«Additional Tools for Xcode»** (свежий, под твою macOS).
   - В скачанном `.dmg` открой `Hardware/Network Link Conditioner.prefPane` — двойной клик ставит его в System Settings.
   - В **System Settings → Network Link Conditioner** появится новая панель с переключателем ON/OFF и списком профилей.
3. Прототип на проде: <https://webrtcvkr.nomorepartiessite.ru> — должен открываться, healthz возвращать «ok».

## Однократная настройка NLC: создание профилей

В System Settings → Network Link Conditioner нажми **Manage Profiles → +** и создай 4 кастомных профиля (профиль «good» — это просто состояние OFF). Параметры берутся из таблицы 3.1 ВКР:

| Имя профиля в NLC | Downlink Bandwidth | Uplink Bandwidth | Downlink Delay | Uplink Delay | Downlink Loss | Uplink Loss |
|---|:---:|:---:|:---:|:---:|:---:|:---:|
| `3g` | 750 kbps | 750 kbps | 100 ms | 100 ms | 0,5 % | 0,5 % |
| `edge` | 250 kbps | 250 kbps | 200 ms | 200 ms | 1 % | 1 % |
| `lossy` | (пусто, без огран.) | (пусто) | 50 ms | 50 ms | 5 % | 5 % |
| `severe` | 100 kbps | 100 kbps | 400 ms | 400 ms | 15 % | 15 % |

> **Важно:** в NLC задержка указывается в одну сторону (uplink + downlink = full RTT). Я поделила значения из таблицы пополам, чтобы суммарный RTT совпал с заявленным в ВКР: для `3g` 200 мс, для `edge` 400 мс, для `lossy` 100 мс, для `severe` 800 мс.

Сохрани профили. NLC по умолчанию выключен.

## Подготовка к эксперименту

1. Закрой все ненужные приложения, особенно те, что могут грузить сеть (Slack, Telegram-десктоп, iCloud sync, торренты, фоновые обновления). Они влияют на телеметрию.
2. Запусти Chrome. Если хочешь чистоту — отдельный пустой профиль или окно в режиме инкогнито.
3. Открой DevTools (F12) на всякий случай, в Console будут видны логи `[app]`, `[collector]` — если что-то не так, я по ним скажу что.

## Один прогон (повторяется 10 раз)

Шаги ниже — для одной комбинации `<профиль> × <режим>`. Всего комбинаций 10: 5 профилей × 2 режима. Подразумевается, что в этот раз тестируем, например, `3g` + `adaptive`.

### Старт сессии

1. Открой <https://webrtcvkr.nomorepartiessite.ru> в **первой** вкладке Chrome.
2. Нажми **«Войти / создать»** — разреши камеру и микрофон, появится локальное видео.
3. Нажми **«Копировать ссылку»**.
4. Открой **новую вкладку** в этом же окне Chrome, вставь ссылку, нажми **Enter**.
5. Во второй вкладке нажми **«Войти / создать»** (она автоматически подцепит room ID из URL). Появится локальное видео и через 1–3 секунды — удалённое.
6. В обеих вкладках в правой панели «Телеметрия WebRTC» должен бежать поток метрик. Состояние ICE = `connected`, состояние PC = `connected`.

### Выбор режима

В **первой** вкладке выбери нужный режим — radio-кнопка `baseline` или `adaptive`. (Достаточно сделать это на одной стороне — клиентский контроллер у каждой стороны свой; для эксперимента важно, что метрики берём именно из вкладки, где идёт запись.)

### Запись телеметрии

1. В первой вкладке в поле справа от текущего profile-инпута впиши идентификатор профиля: `good`, `3g`, `edge`, `lossy`, `severe`.
2. Нажми **«● Старт записи»**. Кнопка станет красной с надписью «⏺ Стоп и скачать», в нижней панели появится «Идёт запись телеметрии.».
3. **Первые 30 секунд (сегмент «pre»)** — ничего не трогай. NLC выключен (OFF). Это базовая линия — нормальные условия перед нагрузкой.
4. Через 30 секунд:
   - Включи NLC: System Settings → Network Link Conditioner → переключатель ON, выбери нужный профиль (например, `3g`).
   - В прототипе нажми **«Mark»** рядом с profile-полем. Это пометит точку смены условий в CSV.
5. **Следующие 60 секунд (сегмент «load»)** — наблюдай. В режиме `adaptive` будут видны переключения политики в строке «Состояние адаптации» дашборда. В `baseline` они не должны срабатывать.
6. Через 60 секунд:
   - Выключи NLC: переключатель OFF (или Manage Profiles → переключи на отсутствующий профиль, чтобы убрать ограничения).
   - В прототипе обнови значение profile-поля на `back-to-good` и нажми **«Mark»**.
7. **Последние 30 секунд (сегмент «post»)** — ничего не трогай, наблюдай восстановление качества.
8. Через 30 секунд нажми **«⏺ Стоп и скачать»**. CSV скачается с автоматическим именем `run-<profile>-<mode>-<timestamp>.csv`.

### Завершение прогона

1. В первой вкладке нажми **«Завершить»**. Соединение закрывается, дашборд показывает финальное состояние, второй пир получает уведомление.
2. Закрой обе вкладки.
3. Перейди к следующей комбинации.

## Полная программа (12 прогонов)

Удобный порядок — менять только один параметр за раз. Сначала все 5 «обычных» профилей в `baseline`, потом все 5 в `adaptive`, в конце — два прогона для сценария `outage` (жёсткий обрыв сети) с разными режимами:

```
1.  baseline + good     →  run-good-baseline-….csv
2.  baseline + 3g       →  run-3g-baseline-….csv
3.  baseline + edge     →  run-edge-baseline-….csv
4.  baseline + lossy    →  run-lossy-baseline-….csv
5.  baseline + severe   →  run-severe-baseline-….csv
6.  adaptive + good     →  run-good-adaptive-….csv
7.  adaptive + 3g       →  run-3g-adaptive-….csv
8.  adaptive + edge     →  run-edge-adaptive-….csv
9.  adaptive + lossy    →  run-lossy-adaptive-….csv
10. adaptive + severe   →  run-severe-adaptive-….csv
11. baseline + outage   →  run-outage-baseline-….csv
12. adaptive + outage   →  run-outage-adaptive-….csv
```

Профиль `good` всё равно прогоняем — это базовая линия для нормальных условий, чтобы в §3.3.1 показать, что прототип «не ломает» обычную работу.

### Особая процедура для профиля `outage`

Этот прогон отличается от других: вместо изменения параметров сети через NLC мы кратковременно полностью отключаем сеть. Шаги:

1. Подключитесь к сессии как обычно (две вкладки, см. «Старт сессии» выше). Дождитесь стабильной связи.
2. Выберите нужный режим (`baseline` или `adaptive`).
3. В поле `profile` впишите `outage`, нажмите **«● Старт записи»**.
4. **30 секунд (сегмент «pre»)** — нормальная работа.
5. **На 30-й секунде:** выключите Wi-Fi на Mac (или дёрните Ethernet-кабель). Видеосвязь оборвётся.
6. **5 секунд** держите сеть выключенной.
7. **На 35-й секунде:** включите Wi-Fi обратно. Mac снова в сети.
8. **Наблюдайте:**
   - В режиме `baseline` ожидается, что связь **не возобновится** автоматически. В дашборде ICE-state будет `failed`, видео не вернётся.
   - В режиме `adaptive` ожидается, что RecoveryManager автоматически инициирует ICE restart, в правом верхнем углу появится тост «Восстанавливаем соединение…», поверх удалённого видео — красный оверлей. Через 3–10 секунд должно появиться «Соединение восстановлено» и видео.
9. **Запишите ещё 60 секунд** после события, чтобы зафиксировать поведение и (для adaptive) измерить время восстановления.
10. Нажмите **«⏺ Стоп и скачать»**.

В режиме `baseline`, если связь не вернулась, после остановки записи нужно завершить сессию через «Завершить» и переподключиться для следующего прогона.

## После прогонов

1. Собери все 10 CSV в одну папку (например, `~/Desktop/webrtc-runs/`).
2. Zip-ни папку или просто перетащи мне файлы в чат.
3. Я прогоню их через анализ-скрипт, получу сводные таблицы и графики, и вставлю в §3.3 главы 3. Параллельно проверю, согласуются ли реальные числа с ожиданиями в §3.4 — если нет, переформулирую обсуждение честно.

## Если что-то не работает

- **Звонок не устанавливается между вкладками** — открой DevTools в одной из вкладок, во вкладке Console посмотри логи `[app]`. Скинь скрин или текст ошибки.
- **Дашборд показывает только нули** — `getStats()` возвращает пусто. Возможно, не дождалась состояния `connected`. Закрой сессию и подключись заново.
- **NLC «не действует»** — проверь, что NLC включён в System Settings и что выбран профиль. Иногда после смены профиля требуется выключить и снова включить переключатель ON.
- **Запись скачалась пустой** — нужно было сначала установить соединение и начать получать метрики, и только потом нажимать «Старт записи».
- **Кнопки «Старт записи» нет в интерфейсе** — обнови страницу Cmd+Shift+R: CSS / JS могли залипнуть в кэше после деплоя.

## Запасной план

Если по какой-то причине Network Link Conditioner отказывается работать (бывает после обновлений macOS), есть резервный вариант — использовать встроенный в macOS `dnctl` / `pfctl`. Дай знать, напишу обходной протокол на этом стеке. Но обычно NLC работает.

**Время на одно прохождение:** запуск ~30 сек + 30 сек pre + 60 сек load + 30 сек post + сохранение ~10 сек ≈ **2,5 минуты на прогон**. Прогоны `outage` чуть длиннее (~3 минуты).

**12 прогонов:** ≈ 30 минут активного времени + 10–15 минут на переключения и проверки = **~45 минут** в сумме.
